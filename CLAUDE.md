## 1. What this is (business)

**AI-Enhanced Web Platform for Research Group Visibility and Impact** — a new web
presence for the **Machine Learning and Knowledge Discovery (MLKD)** group at
**INESC-ID**. It collects the group's publications automatically, summarises them
with AI into plain language, makes them searchable by meaning, and lets
non-technical staff edit everything through an admin panel.

**Primary success criterion (from the brief): the site must be easy to maintain.**
Adding a person, publication, news item or thesis topic must be *easier* than on
the current site. Test every decision with: "does this make maintenance harder?"
If yes, reconsider.

Built by a 4-student Erasmus summer internship team (roles: backend/architecture,
frontend/UX, AI & automation, data & visibility).

## 2. Tech stack

| Layer | Tech | Why |
|---|---|---|
| Site + admin | **Next.js 16 + Payload CMS 3** (TypeScript) | SSR/SEO; the admin panel replaces WordPress — log in, edit a field, done |
| AI service | **Python + FastAPI + LiteLLM** | All AI logic in one service; swap the LLM by changing one env var |
| Database | **PostgreSQL 16 + pgvector** | One DB for relational data *and* vector search |
| Publication data | **OpenAlex API** (+ ORCID / DBLP) | Free, official, has abstracts + citations (Google Scholar has no API) |
| LLM | **Gemini** free tier (swappable) | Summaries / bios / snippets; any provider via LiteLLM |
| Embeddings | **sentence-transformers** (local) | Semantic search, no API cost |
| Deploy | **Docker Compose** | Whole system: `db` + `web` + `ai`, one command |

## 3. Architecture & ownership (hard rules)

Three containers, one `docker compose up`: `web`, `ai`, `db`. **Do not cross these
ownership boundaries:**

- **Payload owns content.** All content tables (publications, members, news, …)
  are created and migrated by Payload. Nothing else writes to them directly.
- **FastAPI owns AI data.** Its own tables (embeddings, topic map). It must
  **not** write to Payload content tables directly — only through the Payload
  REST API (service API key). That is how AI summaries and bio drafts land in the
  admin panel and stay human-editable.
- **Next.js (frontend)** reads content via the Payload Local API (same process);
  calls FastAPI over HTTP for search / map / snippets.

**Why:** a summary stored in a Payload field is editable in the admin ("profile
owners can customise", "easy updates"). Embeddings are not content — nobody edits
them, so they live separately and can be recomputed anytime without clobbering
manual edits. The `aiSummaryStatus=edited` flag protects human edits from being
overwritten by a pipeline. **The LLM is an offline batch step, never a runtime
dependency** — the site works fully even if the LLM provider is down.

## 5. Where to add a feature

So you don't grep the whole repo:

| You want to… | Go to |
|---|---|
| Add a **content type** | `web/src/collections/Xxx.ts` → register in `web/src/payload.config.ts` → add a page in `web/src/app/(frontend)/` (details in `web/CLAUDE.md`) |
| Add a **field** to a content type | edit the collection file → `pnpm generate:types` |
| Add an **AI batch pipeline** | `ai/app/pipelines/<name>.py` (details in `ai/CLAUDE.md`) |
| Add an **AI HTTP endpoint** | `ai/app/api/routes.py` (mutating routes require `X-Service-Token`) |
| Add / change an **LLM prompt** | `ai/app/llm/prompts/<name>.md` — prompts are files, reviewed like code |
| Add a **site page** | `web/src/app/(frontend)/<route>/page.tsx` |
| Change **permissions / roles** | `web/src/access/index.ts` (roles: `admin` / `editor` / `member`) |
| Add / translate a **UI string** | `web/src/i18n/messages.ts` — add the key to **both** `en` and `pt` |

## 6. Global rules

1. **Site languages: English + Português only.** No Russian may ever appear in the
   UI or any user-facing content. The UI is localized via a `NEXT_LOCALE` cookie and
   `web/src/i18n/messages.ts`; a header toggle (`LocaleSwitcher`) switches languages.
   Every user-facing string goes through the dictionary in **both** locales — never
   hardcode visible copy. Bibliographic data from OpenAlex (paper titles, abstracts)
   is not translated. See `web/CLAUDE.md`.
2. **Code, comments, docs and commit messages: English.** (The older Russian docs
   under `docs/` are legacy — do not add new Russian anywhere.)
3. **Comments explain *what*, *why*, and *why this way*** — the intent and the
   trade-off, not a restatement of the code. If a comment just repeats the line
   below it, delete it.
4. **Small, single-purpose files.** When a file outgrows its one job, split it into
   sub-files / folders. Prefer many small, well-bounded modules over a few large
   ones — easier to review and to reason about. A large file is a smell.
5. **Docs-as-structure.** The folder layout *is* the primary documentation. Specs
   and plans live in `docs/`. Keep this file and the nested ones accurate (§7).
6. **Follow existing patterns** before inventing new ones; match the surrounding
   code.

## 7. Keeping CLAUDE.md current (maintenance ritual)

This file has two kinds of content, kept fresh two different ways:

- **The Project map (§4) is auto-generated** by `scripts/gen-claude-map.mjs` and
  refreshed on every commit by `.githooks/pre-commit`. Never edit it by hand.
  (`pnpm install` in `web/` wires the hook up via `scripts/setup-hooks.mjs`.)
- **The prose (everything else) is your responsibility.** **Before finishing any
  task, run the checklist below and update the relevant CLAUDE.md section(s) in the
  same commit.** When in doubt, invoke the `/revise-claude-md` skill — it reviews
  the session's changes and proposes targeted edits.

### CLAUDE.md sync checklist (run before every commit / PR)

- [ ] Added/removed a Payload collection or a notable field? → update `web/CLAUDE.md` + §5
- [ ] Added/removed an AI pipeline or a FastAPI endpoint? → update `ai/CLAUDE.md`
- [ ] Added a new site page/route? → map auto-updates; add a navigation note if non-obvious
- [ ] Changed the architecture or an ownership boundary? → update §3
- [ ] Introduced a new convention or rule? → update §6
- [ ] Added/removed a dependency or env var? → update §2 and `.env.example`
- [ ] Changed *how* a feature is added? → update §5

If none apply, no prose change is needed — the map still auto-updates.

## 8. Setup & run (local)

Requirements: Node ≥ 20.9, pnpm `^9 || ^10`, Python 3.11, Docker. Full details in
`README.md`.

```bash
cp .env.example .env                    # set PAYLOAD_SECRET (openssl rand -hex 32)
docker compose up -d db                 # Postgres + pgvector
cd web && pnpm install && pnpm dev      # http://localhost:3000, admin at /admin
cd ai  && uvicorn app.main:app --reload --port 8000   # http://localhost:8000/docs
```

The local DB starts **empty**; publication data is populated by the `ingest`
pipeline (`ai/`). Payload pushes the schema on first run in dev.
